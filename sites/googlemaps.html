SDK de Maps para Android

Con Maps SDK para Android, se puede agregar mapas basados ​​en datos de Google Maps a la aplicación. La API maneja automáticamente el acceso a los servidores de Google Maps, la descarga de datos, la visualización de mapas y la respuesta a los gestos del mapa. También puede usar llamadas a la API para agregar marcadores, polígonos y superposiciones a un mapa básico y para cambiar la vista del usuario de un área de mapa en particular. Estos objetos proporcionan información adicional para las ubicaciones del mapa y permiten la interacción del usuario con el mapa. La API permite agregar estos gráficos a un mapa:

  • Iconos anclados a posiciones específicas en el mapa ( Marcadores ).
  • Conjuntos de segmentos de línea ( polilíneas ).
  • Segmentos cerrados ( polígonos ).
  • Gráficos de mapa de bits anclados a posiciones específicas en el mapa ( Superposiciones de suelo ).
  • Conjuntos de imágenes que se muestran en la parte superior de los mosaicos del mapa base ( Tile Overlays ).

Accesibilidad

El SDK de Maps para Android incluye compatibilidad integrada para la accesibilidad. Esta sección contiene un resumen de alto nivel de las funciones de accesibilidad que se habilitan automáticamente para cualquier aplicación que utilice la API.

Cuando los usuarios habilitan la función de accesibilidad de TalkBack en sus dispositivos móviles, cada deslizamiento por la pantalla mueve el foco de un elemento de la interfaz de usuario al siguiente. (Una alternativa al deslizamiento único es explorar los elementos de la interfaz de usuario arrastrando un dedo sobre la interfaz). Cuando un elemento de la interfaz de usuario se enfoca, TalkBack lee el nombre del elemento. Si el usuario toca dos veces en cualquier lugar de la pantalla, se realiza la acción enfocada.

Es una buena práctica agregar un anuncio que describa el mapa. Para especificar el texto del anuncio, llame setContentDescription() a la vista.

SDK de Maps para iOS

Al utilizar el SDK, debe cumplir con los Términos de servicio de Google Maps Platform y asegurarse de que su aplicación cumpla con las leyes aplicables. Tenga en cuenta que al utilizar el SDK, el nombre y la versión de su aplicación, la información de autenticación y un identificador anónimo entre aplicaciones se envían automáticamente con cada solicitud.

Requisitos de atribución

Si usa el SDK de Maps para iOS en su aplicación, debe incluir el texto de atribución como parte de una sección de avisos legales en su aplicación. Se recomienda incluir avisos legales como un elemento de menú independiente o como parte de un elemento de menú "Acerca de". Puede obtener el texto de atribución haciendo una llamada a [GMSServices openSourceLicenseInfo].

Plataformas compatibles

Con Maps SDK para iOS, puede crear aplicaciones destinadas a dispositivos nativos de 32 o 64 bits que ejecuten iOS 9.0 y versiones posteriores. El desarrollo de una aplicación con Maps SDK para iOS requiere al menos Xcode 11.7 con un SDK de destino de 11.0 o posterior. (Establecer un SDK de destino de 11.0 o posterior no evitará que su aplicación se ejecute en iOS 9.0).

Las aplicaciones que utilizan las URL de Maps requieren que el dispositivo de destino tenga instalado Google Maps para iOS.

Es poco probable que funcione el SDK de Maps para iOS dentro de las extensiones de la aplicación iOS, debido a las estrictas restricciones de memoria que se aplican.

Maps JavaScript API

La API de JavaScript de Maps le permite personalizar mapas con su propio contenido e imágenes para mostrarlos en páginas web y dispositivos móviles. La API de JavaScript de Maps presenta cuatro tipos de mapas básicos (mapa de ruta, satélite, híbrido y terreno) que puede modificar utilizando capas y estilos, controles y eventos, y varios servicios y bibliotecas.

Cargando la API de JavaScript de Maps

La API de JavaScript de Maps se carga mediante una etiqueta script, que se puede agregar en línea en su archivo HTML o de forma dinámica mediante un archivo JavaScript independiente. Le recomendamos que revise ambos enfoques y elija el que sea más apropiado para cómo está estructurado el código de su proyecto.

La URL en este ejemplo tiene dos parámetros: key donde proporciona su clave de API, y callback, donde especifica el nombre de una función global que se llamará una vez que la API de JavaScript de Maps se cargue por completo. Maps URLs

Con las URL de Maps, se puede crear una URL universal multiplataforma para iniciar Google Maps y realizar búsquedas, obtener direcciones y navegación, y mostrar vistas de mapas e imágenes panorámicas. La sintaxis de la URL es la misma independientemente de la plataforma en uso.

No se necesita una clave API de Google para utilizar las URL de Maps.

Sintaxis universal multiplataforma

Como desarrollador de una aplicación de Android, una aplicación de iOS o un sitio web, puede construir una URL común, que abrirá Google Maps y realizará la acción solicitada, sin importar la plataforma en uso cuando se abra el mapa.

En un dispositivo Android:
  • Si la aplicación Google Maps para Android está instalada y activa, la URL inicia Google Maps en la aplicación Maps y realiza la acción solicitada.
  • Si la aplicación Google Maps no está instalada o está desactivada, la URL inicia Google Maps en un navegador y realiza la acción solicitada.
En un dispositivo iOS:
  • Si la aplicación Google Maps para iOS está instalada, la URL inicia Google Maps en la aplicación Maps y realiza la acción solicitada.
  • Si la aplicación Google Maps no está instalada, la URL inicia Google Maps en un navegador y realiza la acción solicitada.
  • En cualquier otro dispositivo, la URL inicia Google Maps en un navegador y realiza la acción solicitada.

API Places para Web

Es un servicio que devuelve información sobre lugares mediante solicitudes HTTP. Los lugares se definen dentro de esta API como establecimientos, ubicaciones geográficas o puntos de interés destacados.


Se accede a cada uno de los servicios como una solicitud HTTP y devuelve una respuesta JSON o XML. La API Places utiliza un ID de lugar para identificar un lugar de forma única.

Están disponibles las siguientes solicitudes de lugar:

  • Búsqueda de lugar
  • Detalles de lugar
  • Fotos de lugar
  • Autocompletar
  • Autocompletar consultas


Búsqueda de lugar

Permite buscar información de lugares utilizando una variedad de categorías, incluidos establecimientos, puntos de interés destacados y ubicaciones geográficas. Puede buscar lugares por proximidad o por una cadena de texto.


Una solicitud Find Place toma una entrada de texto y devuelve un lugar. La entrada puede ser cualquier tipo de datos de texto de lugares, como un nombre, una dirección o un número de teléfono, en caso de que los datos que no sean cadenas, como latitud o longitud, se generará un error.

Una solicitud Find Place es una URL HTTP con el siguiente formato:

Parámetros requeridos:

  • key: Clave API de su aplicación
  • input: Entrada de texto que identifica el objetivo de la búsqueda, como un nombre, dirección o número de teléfono.
  • inputtype: El tipo de entrada. Este puede ser uno de textquery o phonenumber.

Una solicitud de búsqueda cercana es una URL HTTP con el siguiente formato:

Parámetros requeridos:

  • key: Clave API de su aplicación.
  • location: La latitud / longitud alrededor de la cual recuperar la información del lugar. Esto debe especificarse como latitud, longitud.
  • radius: Define la distancia (en metros) dentro de la cual devolver los resultados del lugar. El radio máximo permitido es de 50 000 metros.


Las solicitudes de búsqueda de texto es un servicio web que devuelve información sobre un conjunto de lugares en función de una cadena, por ejemplo, "pizza en Nueva York" o "zapaterías cerca de Ottawa" o "123 Main Street". El servicio responde con una lista de lugares que coinciden con la cadena de texto y cualquier sesgo de ubicación que se haya establecido.

El servicio es especialmente útil para realizar consultas de direcciones ambiguas en un sistema automatizado, y los componentes de la cadena que no son direcciones pueden coincidir con empresas y direcciones. Ejemplos de consultas de direcciones ambiguas son direcciones incompletas, direcciones con formato deficiente o una solicitud que incluye componentes que no son direcciones, como nombres comerciales.


La respuesta a la búsqueda incluirá una lista de lugares. Puede enviar una solicitud de detalles del lugar para obtener más información sobre cualquiera de los lugares en la respuesta.

Una solicitud de búsqueda de texto es una URL HTTP con el siguiente formato:

  • query: La cadena de texto en la que buscar, por ejemplo: "restaurante" o "123 Main Street". El servicio devolverá coincidencias de candidatos en función de esta cadena y ordenará los resultados según su relevancia percibida.
  • key: Clave API de su aplicación.

Autocompletar

El servicio Place Autocomplete devuelve predicciones de lugares en respuesta a una solicitud HTTP. La solicitud especifica una cadena de búsqueda textual y límites geográficos opcionales. El servicio se puede utilizar para proporcionar la funcionalidad de autocompletar para búsquedas geográficas basadas en texto, devolviendo lugares como negocios, direcciones y puntos de interés a medida que el usuario escribe.

El servicio puede hacer coincidir palabras completas y subcadenas, resolviendo nombres de lugares, direcciones y códigos más. Por lo tanto, las aplicaciones pueden enviar consultas a medida que el usuario escribe, para proporcionar predicciones de lugares sobre la marcha.

Las predicciones devueltas están diseñadas para ser presentadas al usuario para ayudarlo a seleccionar el lugar deseado.

API de geolocalización

La API de geolocalización devuelve una ubicación y un radio de precisión basados en información sobre torres de telefonía celular y nodos WiFi que el cliente móvil puede detectar.

Las solicitudes de geolocalización se envían mediante POST a la siguiente URL:

El cuerpo de la solicitud debe tener el formato JSON. Se admiten los siguientes campos y todos los campos son opcionales:

  • homeMobileCountryCode: El código de país móvil (MCC) de la red doméstica del dispositivo.
  • homeMobileNetworkCode: El código de red móvil (MNC) de la red doméstica del dispositivo.
  • radioType: El tipo de radio móvil. Los valores admitidos son lte, gsm, cdma, y wcdma. Si bien este campo es opcional, debe incluirse si hay un valor disponible para obtener resultados más precisos.
  • carrier: El nombre del operador.
  • considerIp: Especifica si se debe recurrir a la geolocalización de IP si las señales de torre celular y wifi no están disponibles. Por defecto es true. Configure considerIp en false para deshabilitar el retroceso.
  • cellTowers: Una serie de objetos de torres de telefonía móvil.
  • wifiAccessPoints: Una serie de objetos de punto de acceso WiFi.

La cellTowersmatriz del cuerpo de la solicitud contiene cero o más objetos de torre celular.

  • cellId(obligatorio): Identificador único de la celda. En GSM, este es el ID de celda (CID); Las redes CDMA utilizan el ID de la estación base (BID). Las redes WCDMA utilizan la identidad de celda UTRAN / GERAN (UC-Id), que es un valor de 32 bits que concatena el controlador de red de radio (RNC) y la ID de celda. Especificar solo el valor de ID de celda de 16 bits en redes WCDMA puede generar resultados inexactos.
  • locationAreaCode(obligatorio para radioType gsmy cdma, de lo contrario opcional): el código de área de ubicación (LAC) para redes GSM y WCDMA. El ID de red (NID) para redes CDMA.
  • mobileCountryCode (obligatorio): el código de país móvil (MCC) de la torre de telefonía móvil.
  • mobileNetworkCode(obligatorio): el código de red móvil de la torre de telefonía móvil. Este es el MNC para GSM y WCDMA; CDMA usa el ID del sistema (SID).

La wifiAccessPoints matriz del cuerpo de la solicitud debe contener dos o más objetos de punto de acceso WiFi.

  • macAddress: ( obligatorio ) La dirección MAC del nodo WiFi. Por lo general, se llama dirección BSS, BSSID o MAC. Los separadores deben ser :(dos puntos).
  • signalStrength: La intensidad de la señal actual medida en dBm.
  • age: El número de milisegundos desde que se detectó este punto de acceso.
  • channel: El canal a través del cual el cliente se comunica con el punto de acceso.
  • signalToNoiseRatio: La relación señal / ruido actual medida en dB.

API de zona horaria

La API de zona horaria proporciona una interfaz sencilla para solicitar la zona horaria de las ubicaciones en la superficie de la tierra, así como la diferencia horaria de UTC para cada una de esas ubicaciones. Solicita la información de la zona horaria para una fecha y un par de latitud / longitud específicos. La API devuelve el nombre de esa zona horaria, la diferencia horaria de UTC y la diferencia de horario de verano.

Las solicitudes de la API de zona horaria se construyen como una cadena de URL.Se debe tener en cuenta que es posible que los datos de la zona horaria no estén disponibles para ubicaciones sobre el agua, como océanos o mares.

Una solicitud de API de zona horaria tiene la siguiente forma:

Parámetros requeridos:

  • location: una tupla lat, lng separada por comas (p. ej. location=-33.86,151.20), que representa la ubicación para buscar.
  • timestamp: especifica el tiempo deseado en segundos desde la medianoche del 1 de enero de 1970 UTC. La API de zona horaria utiliza timestamp para determinar si se debe aplicar o no el horario de verano, según la zona horaria de location. Tenga en cuenta que la API no tiene en cuenta las zonas horarias históricas. Es decir, si especificas un pasado timestamp, la API no tiene en cuenta la posibilidad de que location estuviera anteriormente en una zona horaria diferente.
  • key: Clave API de su aplicación.
